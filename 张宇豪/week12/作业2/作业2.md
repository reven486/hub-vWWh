# 1. 前后端分离

在 06-stock-bi-agent 中，Streamlit作为前端，FastAPI作为后端，MCP作为中间件。这三个组件通过网络协议（HTTP/SSE）相互通信，但逻辑上是完全独立的。

Streamlit负责和用户交互，接收用户的输入，并传给后端，再将后端返回的数据经过渲染后返回给用户。

FastAPI负责处理逻辑和加工数据，进行具体的功能实现。

MCP负责实现“特殊技能”，当后端 或者 LLM 发现需要查天气、查阅新闻、查询股票时，它不自己写这些逻辑，而是通过MCP协议去调用这些外部工具。

好处：

（1）可以进行解耦和独立演进：如果以后想将 Streamlit 换成 Vue 或 React，只需要重新写前端页面即可，后端的接口不需要修改。

（2）并发与扩展性：FastAPI 可以同时处理多个 Streamlit 用户的请求。MCP可以单独作为一个服务器部署在另一个机器上，专门实现“特殊技能”，而不影响界面的流畅性。





# 2. 如何实现历史对话

### `chat_list.py`

该文件主要负责展示用户过去创建的所有聊天记录。

- **获取列表**：通过向后端 `http://127.0.0.1:8000/v1/chat/list` 发送请求，获取当前用户（`user_name`）的所有 `session_id` 和会话标题（`title`）。
- **进入历史对话**：当点击“进入聊天”按钮时，代码会将选中的 `session_id` 存入 `st.session_state`，并跳转到 `chat/chat.py` 页面。
- **管理功能**：实现了删除历史对话的功能（调用 `/v1/chat/delete`）。

### `chat.py`

该文件在初始化时会检查是否存在 `session_id`，从而决定是开启新对话还是恢复旧对话。

```python
if "session_id" in st.session_state.keys() and st.session_state.session_id:
    data = requests.post("http://127.0.0.1:8000/v1/chat/get?session_id=" + st.session_state['session_id']).json()
    for message in data["data"]:
        if message["role"] == "system":
            continue
        st.session_state.messages.append({"role": message["role"], "content": message["content"]})
```

- 如果 `session_state` 中已经有了 `session_id`（从列表页跳转过来），它会向后端请求该会话的所有历史消息（`/v1/chat/get`），并填充到当前页面的消息流中。
- **界面渲染**：随后通过循环 `for message in st.session_state.messages` 将这些历史记录渲染在 Streamlit 界面上。

### 3. 对话持久化逻辑

- **新会话初始化**：如果没有 `session_id`，在用户发送第一条消息时，会调用 `request_session_id()`（请求 `/v1/chat/init`）来生成一个新的 `session_id`。
- **后续发送**：在 `request_chat` 函数中，每次发送新消息都会带上 `session_id`，这确保了后端可以将新的对话内容持续追加到数据库中的该会话下。
